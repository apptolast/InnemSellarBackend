5//! Backend de InemSellar — API REST para la app de ayuda a desempleados en Espana.
6//!
7//! # Arquitectura
8//! Este crate usa [Salvo](https://salvo.rs) como framework web, SeaORM como ORM
9//! para PostgreSQL, y JWT para autenticacion. Cada modulo incluye documentacion
10//! educativa que explica los conceptos de Rust utilizados.

12//! # Documentacion de la API

14//! - Swagger UI: `http://localhost:8080/swagger-ui`
15//! - JSON OpenAPI: `http://localhost:8080/api-doc/openapi.json`
16//!
17//! # Como generar la documentacion Rust en HTML

40/// Handler basico que responde con "Hello World".
41///
42/// # Por que `#[handler]`
43/// `#[handler]` es una macro de Salvo que convierte una funcion async en un handler HTTP.
44/// Gestiona automaticamente la extraccion de parametros de la request y la serializacion
45/// de la respuesta. Es similar a las anotaciones de ruta en frameworks como Shelf en Dart.
46///
47/// # Por que `#[handler]` aqui y no `#[endpoint]`
48/// Este handler de health-check es tan simple que no aporta valor documentarlo en OpenAPI.
49/// `#[endpoint]` se reserva para los handlers de negocio que necesitan documentacion.
50/// Tecnicamente ambos funcionan igual en runtime; la diferencia es solo en que
51/// `#[endpoint]` genera metadata OpenAPI y `#[handler]` no.
52///
53/// # Por que `&'static str`
54/// `'static` es un lifetime que indica que el string vive durante toda la ejecucion del programa.
55/// Los literales de string en Rust (`"Hello World"`) siempre son `&'static str` porque se
56/// almacenan en el binario compilado, no en el heap. No requieren allocacion ni liberacion.
57#[handler]
58async fn hello() -> &'static str {
59    "Hello World"
60}
61
62/// Punto de entrada de la aplicacion.
63///
64/// # Por que `#[tokio::main]`
65/// Rust no tiene async runtime integrado (a diferencia de Dart que tiene su event loop).
66/// `#[tokio::main]` inicializa el runtime async de tokio, que gestiona las tareas
67/// concurrentes (conexiones HTTP, queries a DB, etc.). Es como arrancar el event loop
68/// de Dart, pero de forma explicita.
69///
70/// # Por que `async fn main()`
71/// `async` permite usar `.await` dentro de la funcion. Cada `.await` cede el control
72/// al runtime para que pueda ejecutar otras tareas mientras esperamos I/O (red, disco).
73/// Igual que en Dart con `async/await`, pero en Rust el compilador verifica en tiempo
74/// de compilacion que no haya data races.
75#[tokio::main]
76async fn main() {
77    // Cargamos la configuracion antes de inicializar el logger para que
78    // DATABASE_URL y SERVER_ADDR esten disponibles. `from_env()` hace panic
79    // con mensaje claro si faltan variables obligatorias — fallo rapido
80    // intencionado en arranque.
81    let cfg = config::AppConfig::from_env();
82
83    // Inicializamos el sistema de logging estructurado.
84    // `fmt()` configura el formato de salida (texto legible por humanos).
85    // `with_env_filter` lee la variable de entorno RUST_LOG para decidir
86    // que niveles mostrar (ej: RUST_LOG=info o RUST_LOG=debug).
87    // Si RUST_LOG no esta definida, no se muestra ningun log.
88    // Equivale a configurar el nivel de log en Flutter con `Logger.root.level`.
89    tracing_subscriber::fmt()
90        .with_env_filter(tracing_subscriber::EnvFilter::from_default_env())
91        .init();
92
93    // Creamos la conexion a PostgreSQL via SeaORM.
94    // SeaORM gestiona el pool internamente (usa SQLx por debajo).
95    let db = db::init_db(&cfg).await;
96
97    // Creamos servicios y repositorios, inyectando la conexion.
98    let auth_service = AuthService::new(cfg.jwt_secret.clone(), cfg.jwt_expiracion_minutos);
99    let geo_repo = SeaGeografiaRepo::new(db.clone());
100    let auth_repo = SeaAuthRepo::new(db.clone());
101    let oferta_repo = SeaOfertaRepo::new(db.clone());
102
103    // Inyectamos todos los servicios y repos en el Depot de Salvo.
104    //
105    // # Por que inyectar en el router raiz
106    // `affix_state::inject(valor)` es un middleware que inserta el valor en el `Depot`
107    // de cada peticion. Al ponerlo en el router raiz con `.hoop(...)`, todos los
108    // handlers hijo pueden acceder a el con `depot.obtain::<Tipo>()`.
109    // Es el patron de "inyeccion de dependencias" de Salvo: similar a los Providers
110    // de Riverpod en Flutter, pero sin arbol de widgets — solo un mapa tipado por request.
111    let router = Router::new()
112        .get(hello)
113        .hoop(affix_state::inject(auth_service))
114        .hoop(affix_state::inject(geo_repo))
115        .hoop(affix_state::inject(auth_repo))
116        .hoop(affix_state::inject(oferta_repo))
117        .push(routes::crear_router());
118
119    // ── Documentacion OpenAPI / Swagger UI ──────────────────────────────────
120    //
121    // # Por que OpenAPI
122    // OpenAPI (antes Swagger) es el estandar de la industria para documentar APIs REST.
123    // El spec JSON describe todos los endpoints, parametros, esquemas de request/response
124    // y requisitos de autenticacion. Swagger UI lo convierte en una interfaz web interactiva
125    // donde los desarrolladores (y el equipo de Flutter) pueden explorar y probar la API.
126    //
127    // # Como funciona con Salvo OAPI
128    // `OpenApi::new(...)` recorre el router y recopila la metadata que generaron los
129    // `#[endpoint]` (tags, parametros, esquemas de DTOs marcados con `ToSchema`,
130    // esquemas de seguridad). El resultado es un JSON valido segun la spec OpenAPI 3.x.
131    //
132    // `.merge_router(&router)` es el paso clave: Salvo inspecciona el arbol de routers
133    // buscando handlers marcados con `#[endpoint]` y extrae su informacion de tipos
134    // para construir el spec OpenAPI automaticamente.
135    //
136    // # Por que `add_security_scheme("bearer_auth", ...)`
137    // Define el esquema de autenticacion JWT que referencian los endpoints con
138    // `security(("bearer_auth" = []))`. Le dice a OpenAPI: "hay un esquema llamado
139    // bearer_auth que es HTTP Bearer con formato JWT". Swagger UI mostrara un boton
140    // "Authorize" donde el usuario puede pegar su JWT para probar endpoints protegidos.
141    let doc = OpenApi::new("InemSellar API", "0.1.0")
142        .info(
143            Info::new("InemSellar API", "0.1.0")
144                .description(
145                    "API REST del backend de InemSellar\n\n\
146                     App de ayuda a desempleados en Espana — SEPE/INEM.\n\n\
147                     **Autenticacion**: usa `POST /api/v1/auth/login` para obtener un \
148                     `access_token` JWT y pulsalo en el boton Authorize (arriba a la derecha).",
149                )
150                .contact(
151                    salvo::oapi::Contact::new()
152                        .name("AppToLast")
153                        .email("admin@apptolast.com"),
154                ),
155        )
156        .add_security_scheme(
157            "bearer_auth",
158            // `Http` define un esquema de autenticacion HTTP estandar.
159            // `HttpAuthScheme::Bearer` indica que el token va en el header
160            // `Authorization: Bearer <token>`. `.bearer_format("JWT")` es
161            // solo un hint para Swagger UI (no afecta a la validacion).
162            SecurityScheme::Http(Http::new(HttpAuthScheme::Bearer).bearer_format("JWT")),
163        )
164        .merge_router(&router);
165
166    // Anadimos las rutas de documentacion al router existente.
167    //
168    // # Por que `.unshift(...)` y no `.push(...)`
169    // `.unshift()` inserta el router al PRINCIPIO del arbol de rutas, antes que las
170    // rutas de negocio. Esto es necesario para que las rutas `/api-doc/openapi.json`
171    // y `/swagger-ui` se registren correctamente sin conflictos con el prefijo `/api/v1`.
172    //
173    // # Las dos rutas de documentacion
174    // - `/api-doc/openapi.json`: el spec JSON bruto (util para herramientas externas)
175    // - `/swagger-ui`: interfaz web interactiva (para humanos)
176    let router = router
177        .unshift(doc.into_router("/api-doc/openapi.json"))
178        .unshift(SwaggerUi::new("/api-doc/openapi.json").into_router("/swagger-ui"));
179
180    // `TcpListener` abre el socket TCP en el puerto 8080 en todas las interfaces
181    // de red (0.0.0.0). `.bind().await` completa el binding de forma asincrona.
182    // En produccion, Nginx actua como proxy inverso delante de este puerto.
183    let acceptor = TcpListener::new(cfg.port_addr).bind().await;
184
185    tracing::info!("Servidor escuchando en http://{}", cfg.server_addr);
186    tracing::info!(
187        "Documentacion Swagger UI: http://{}/swagger-ui",
188        cfg.server_addr
189    );
190
191    // `Server::new(acceptor).serve(router).await` arranca el bucle principal
192    // de aceptacion de conexiones. Esta llamada no retorna hasta que el proceso
193    // recibe una senal de terminacion (SIGTERM/SIGINT). Es el equivalente al
194    // `runApp(MyApp())` en Flutter: el punto sin retorno que cede el control
195    // al framework.
196    Server::new(acceptor).serve(router).await;
197}
